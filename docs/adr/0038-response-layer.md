# ADR 0038: Response Protection Layer

**Date:** 2026-04-15
**Status:** Implemented
**Deciders:** GuardianWAF Team

---

## Context

WAF protection is not limited to blocking inbound attacks — the response must also be hardened:

- **Security headers** (CSP, HSTS, X-Frame-Options) reduce the browser's attack surface; they must be consistently present on every response, including error responses and redirects
- **Stack traces and debug information** in error responses reveal implementation details (framework version, file paths, SQL queries) that aid attackers
- **Sensitive data in responses** (API keys, SSNs, credit card numbers in debug logs or error messages) must be masked before reaching clients
- **Branded block pages** — when the WAF blocks a request, the 403 response should be informative and branded, not a generic nginx or Go error page

These concerns span every response, including those generated by the WAF itself (block pages, challenge pages). A single response layer (Order 600, the last layer) ensures uniform application.

## Decision

Implement a response protection layer (`internal/layers/response/`, Order 600) that:

1. **Injects security headers** into every response
2. **Strips stack traces** from error responses
3. **Masks sensitive data** found in response bodies (first-pass detection; deep DLP is handled by ADR 0020)
4. **Renders branded block pages** when the WAF issues a block action

### Security Headers

Default security headers injected on every response:

| Header | Default Value | Rationale |
|--------|-------------|-----------|
| `X-Content-Type-Options` | `nosniff` | Prevent MIME sniffing |
| `X-Frame-Options` | `SAMEORIGIN` | Clickjacking protection |
| `X-XSS-Protection` | `1; mode=block` | Legacy XSS filter (deprecated but harmless) |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Limit referrer leakage |
| `Permissions-Policy` | `geolocation=(), camera=(), microphone=()` | Disable powerful features |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Force HTTPS |
| `Content-Security-Policy` | configurable | Script/style origin control |

**HSTS** is only injected when the connection is HTTPS (checked via `ctx.TLS`). Injecting HSTS on HTTP connections would trigger browser warnings.

**CSP** is configurable per-tenant. A default restrictive CSP is provided but operators commonly need to customize it to allow their CDN and analytics domains. When the client-side protection layer (ADR 0021) is active, the agent's `nonce` value is automatically injected into the CSP `script-src` directive.

Headers are injected by setting `ctx.Metadata["response_headers_hook"]` to a callback function. The engine's `Middleware` calls this hook after writing the backend response headers but before sending to the client. This avoids buffering the response body.

### Stack Trace Stripping

For responses with `Content-Type: text/html` or `application/json` and status codes 4xx/5xx, the layer scans the response body for stack trace patterns:

```
Patterns stripped:
  - Go: "goroutine \d+ \[running\]:"
  - Java: "at com\.\w+\.\w+\..*\(.*\.java:\d+\)"
  - Python: "Traceback (most recent call last):"
  - PHP: "#\d+ \/.*\.php\(\d+\)"
  - .NET: "   at System\.\w+.*"
  - SQL errors: "ORA-\d+", "SQLSTATE\[", "mysql_fetch"
```

Matched sections are replaced with a generic message (`[error details hidden]`). This is a heuristic scan — it may miss obfuscated stack traces or new framework formats. The DLP layer (ADR 0020) handles more structured sensitive data.

### Sensitive Data Masking

A lightweight pass (distinct from the full DLP engine) masks:
- Credit card numbers matching a Luhn-valid 13–19 digit pattern → `****-****-****-XXXX`
- SSN patterns `XXX-XX-XXXX` → `***-**-XXXX`
- API key patterns (`sk_live_...`, `AKIA...`, `ghp_...`) → `[REDACTED]`

This first-pass masking catches the most common cases in error responses without the full DLP pattern engine overhead. When the DLP layer is enabled, it handles the complete masking task; this layer's masking can be disabled via `DataMaskingEnabled: false` to avoid double-processing.

### Branded Block Pages

When the WAF blocks a request (`ActionBlock`), the engine calls `blockpage.Render(ctx)` to generate the 403 response. The `errorpage.go` file provides:

- **Production mode**: minimal page with a reference ID (from `ctx.RequestID`), no technical details
- **Development mode**: includes the triggering rule ID, score, and WAF findings for debugging

The block page template is configurable — operators can inject their own HTML/CSS branding via `error_page_template_path`. Variables available in the template: `{{.RequestID}}`, `{{.Score}}`, `{{.Action}}`, `{{.Contact}}`.

### Configuration

```yaml
response:
  security_headers:
    enabled: true
    hsts_max_age: 31536000
    hsts_subdomains: true
    x_frame_options: SAMEORIGIN
    content_security_policy: "default-src 'self'"
    referrer_policy: "strict-origin-when-cross-origin"
    permissions_policy: "geolocation=(), camera=()"

  data_masking:
    enabled: true
    mask_credit_cards: true
    mask_ssn: true
    mask_api_keys: true
    strip_stack_traces: true

  error_page:
    mode: production             # production | development
    template_path: ""            # Empty = built-in template
    contact_email: "security@example.com"
```

### Per-Tenant Overrides

Tenants can disable security headers (e.g., a tenant whose application manages its own CSP) or override the error page template via `TenantWAFConfig.Response`. Header values configured by the tenant **replace** (not append to) the global defaults.

## Consequences

### Positive
- Security headers are guaranteed to be present on every response — no exception for error responses, redirects, or WAF-generated pages
- Stack trace stripping eliminates a common source of information disclosure without requiring backend changes
- Branded block pages improve user experience and reduce confusion compared to raw WAF error pages

### Negative
- Response body scanning for stack traces and masking requires buffering the response — this conflicts with streaming responses (Server-Sent Events, chunked transfer) where buffering introduces latency
- Stack trace pattern matching is heuristic; novel framework error formats are missed until patterns are updated
- Setting `Strict-Transport-Security` at the WAF level means the HSTS policy applies to all paths including non-WAF-protected paths on the same domain — misconfiguration can lock out HTTP fallback for months
